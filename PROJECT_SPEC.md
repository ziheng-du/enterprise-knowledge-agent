## 一、项目背景与目标

请帮我开发一个 **"企业新员工知识助手"** 项目,这是一个融合了 RAG(检索增强生成)和 Agent(智能体自主决策)能力的 Python 应用。

**业务场景**:模拟企业内部知识库场景,员工可以通过对话方式查询公司规章制度、请假政策、报销流程等内容(RAG 检索),同时对于需要"计算"或"实时查询"类的问题(比如"我入职3年了,年假还剩多少天"),系统应能自主判断并调用相应工具完成任务(Agent 能力),而不是单纯基于文本检索给出模糊回答。

**项目定位**:这是一个用于校招/实习求职的作品集项目,代码质量、架构清晰度、可扩展性和文档完整度,与功能本身同等重要。请始终以"可维护、可扩展、可讲解"为标准来写代码,而不是"能跑就行"。

## 二、技术栈(请严格按此选型,不要擅自替换)

- **语言**:Python 3.10+
- **Agent/编排框架**:LangGraph(用于构建基于状态图的 Agent 决策流程,不要用简单的 if-else 硬编码逻辑代替)
- **RAG 框架**:LangChain(用于文档加载、切分、检索链)
- **向量数据库**:Chroma(本地持久化模式,不依赖云服务)
- **Embedding 模型**:优先使用开源本地模型(如 BAAI/bge-small-zh 或 text2vec 系列,中文效果较好);如果调用在线 Embedding API,需在配置文件中可切换
- **LLM**:通过配置文件支持切换(默认接入 OpenAI 兼容接口或国内大模型 API,如通义千问/DeepSeek,需支持通过 `.env` 文件配置 API_KEY 和 BASE_URL,不要硬编码密钥)
- **后端 Web 框架**:FastAPI(选择理由:原生支持 async/await,适合 LLM 调用与向量检索这类 I/O 密集型场景;天然支持流式输出 `StreamingResponse`,方便做打字机效果的对话流式返回;自带 Swagger UI 交互式文档,便于演示接口)
- **前端**:不需要复杂框架,用简单的 HTML + JS(可以用一个单页面 + fetch 调用后端 API 实现聊天界面),风格简洁清爽即可,不追求华丽
- **日志**:使用 Python 标准库 `logging`,不要用 `print` 调试
- **依赖管理**:提供 `requirements.txt`

## 四、核心功能模块详细要求

### 4.1 文档处理与切分(chunking.py)

- 需要实现 **至少两种切分策略**,并可通过配置切换:
  1. 固定长度切分(fixed-size chunking,带 overlap)
  2. 按语义/段落结构切分(recursive character splitting,优先在段落/句子边界切分)
- 每种策略需要可配置 `chunk_size` 和 `chunk_overlap` 参数
- 切分时需保留原始文档的元数据(来源文件名、页码/段落位置),方便后续在回答中标注引用来源

### 4.2 检索模块(retriever.py)

- 支持相似度检索(similarity search),返回结果需带相似度分数
- 支持配置 `top_k` 和相似度阈值(低于阈值的结果应过滤掉,避免不相关内容被强行塞给 LLM)
- **这个模块要完全独立于 Agent 逻辑**,即它应该可以被单独调用测试,不依赖 LangGraph 的上下文

### 4.3 工具层设计(重点 —— 这是为后续扩展 MCP 预留的架构)

请在 `tools/base.py` 中定义一个**统一的工具抽象基类**,大致设计如下(请你根据 LangChain/LangGraph 的工具调用规范来完善细节,但必须包含以下要素):

- 每个工具必须声明:`name`(工具名)、`description`(供 LLM 判断何时调用的自然语言描述)、`input_schema`(结构化输入参数定义,建议用 Pydantic BaseModel)、`run()` 方法(实际执行逻辑)
- 所有工具通过 `registry.py` 统一注册和管理,Agent 节点通过 registry 动态获取可用工具列表,**不要在 Agent 决策逻辑里直接 import 具体工具函数**
- 这样设计的目的:未来某个工具需要改造成 MCP Server 暴露给外部时,只需要在工具外面包一层协议适配层,核心业务逻辑(`run()` 内部实现)不需要改动。请在代码注释中体现这个设计意图。

**至少实现两个工具**:
1. `leave_calculator`:输入入职日期,计算当前可用年假天数(可以自己设定一个简单的年假规则,比如"每满1年+1天,上限15天",在代码注释里写清楚规则来源是模拟设定)
2. `policy_lookup`:根据关键词查询结构化的制度参数(比如报销额度、审批流程节点),这个数据可以用一个简单的本地 JSON/字典模拟,不需要真实数据库

### 4.4 Agent 决策流程(agent/graph.py)

- 使用 LangGraph 构建一个 **ReAct 模式**的状态图,大致流程:
  1. 接收用户输入
  2. LLM 判断:这个问题是"纯知识检索类"(走 RAG)还是"需要工具计算/查询类"(走工具调用),也可能需要两者结合
  3. 如果需要检索,调用 retriever 获取相关文档片段,拼接进 context
  4. 如果需要工具,自主选择合适的工具并生成调用参数,执行后获取结果
  5. 综合信息生成最终回答
- **必须实现错误处理与兜底逻辑**,例如:
  - 工具调用参数解析失败时,如何重试或降级为直接用 LLM 回答并提示"信息可能不准确"
  - 检索不到相关文档时,应明确告知用户"知识库中未找到相关信息",而不是让 LLM 编造答案(需要在 Prompt 中明确约束)
  - 设置最大工具调用轮次(比如最多 3 轮),避免死循环
- 请在 `state.py` 中清晰定义 Agent 的状态字段(如:用户输入、检索结果、工具调用历史、当前轮次、最终回答等)

### 4.5 API 层(api/routes.py)

- 使用 FastAPI 的 `APIRouter` 组织路由,请求/响应体用 Pydantic BaseModel 定义(这样能自动生成清晰的 Swagger 文档,也是 FastAPI 的核心优势,请充分利用)
- 提供 `POST /api/chat` 接口,接收用户消息,返回 Agent 的回答
- 返回结果中除了最终答案文本,建议附带:是否使用了检索(及来源文档片段)、是否调用了工具(及调用了哪个工具),这样前端可以展示"思考过程",增强项目的展示效果
- **建议额外实现一个流式版本**:`POST /api/chat/stream`,使用 FastAPI 的 `StreamingResponse` 返回逐字/逐token的流式回答(类似 ChatGPT 打字机效果)。如果时间不够,这个可以放在阶段五完成核心功能后再做,作为加分项
- 提供 `POST /api/ingest` 接口(或对应的 CLI 脚本),用于触发文档重新入库
- 启动方式使用 `uvicorn` 运行(请在 README 中写清楚启动命令,如 `uvicorn app.main:app --reload`)

### 4.6 前端(简洁即可,不是重点)

- 一个简单的聊天界面,输入框 + 对话气泡展示
- 如果 API 返回了"使用了哪个工具"或"检索来源",在界面上用小标签展示出来(比如回答下方显示"📄 来源:员工手册.pdf" 或 "🔧 调用了工具:年假计算器"),这个细节能显著提升演示效果
