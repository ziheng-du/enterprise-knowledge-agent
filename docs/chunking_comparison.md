# Chunking 策略对比记录（阶段二选型依据）

测试时间：2026-07-07
测试方式：分别用 `python scripts/ingest.py --rebuild --strategy fixed / recursive` 入库
（chunk_size=500, chunk_overlap=50，两种策略均切出 8 个文本块），
再用 `python scripts/search_demo.py "<问题>" --top-k 2` 对比命中结果。
Embedding 模型：BAAI/bge-small-zh-v1.5（本地，余弦相似度，分数区间 0-1）。

## 定量结果（top-1 命中）

| 测试查询 | fixed 分数 / 来源 | recursive 分数 / 来源 |
|---|---|---|
| 报销多久内要提交 | 0.6898 报销制度.md | 0.6962 报销制度.md |
| 入职3年年假几天 | 0.7284 请假与年假政策.md | 0.7311 请假与年假政策.md |
| 迟到超过30分钟怎么处理 | 0.6160 员工手册.md | 0.5679 员工手册.md |

两种策略的 top-1 命中文档全部正确，recursive 在 3 个查询中 2 个分数略高。

## 定性观察（块边界质量，差异更关键）

- **fixed 存在句中截断**：如报销制度块 1 从"员工在费用系统填写报销单…"开始，
  丢失了"三、报销流程"标题；请假政策块 1 开头是残句"…工资。3. 全年事假累计…"。
  这类残块作为 LLM 上下文时缺少语境，引用展示的可读性也差。
- **recursive 块边界全部落在标题/段落处**：每块自带章节标题，语义完整，
  更适合直接拼进 Prompt 与在前端展示引用。
- 查询 3 中 fixed 分数更高属于偶然：固定切分恰好把"迟到超过 30 分钟"规则
  切到了块首，查询词密度变高；但同样的机制在其他查询中产生的是有害残句。

## 结论

**默认采用 recursive（递归字符切分）策略**，配置项 `CHUNKING_STRATEGY=recursive`
保持为默认值。fixed 策略保留作为对比基线，可随时通过配置或
`--strategy fixed` 切换复现本实验。
